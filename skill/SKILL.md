# draw.io スイムレーン業務フローチャート生成スキル

## 概要

顧客との打合せや業務プロセス定義で使用する **縦型スイムレーン（部門横断）フローチャート** を
draw.io XML形式で生成する。各レーン（部門/役割）ごとにプロセスノード、判断ノード、
開始/終了ノードなどを配置し、エッジ（矢印）で接続する。

議事録やテキスト説明など、フロー構造が明示されていない入力に対しては、
**フェーズ1（分析・整理）** を必ず先に実行し、ユーザーの確認を経てからXML生成に進む。

---

## フェーズ1：ワークフロー分析・整理

テキスト（議事録・業務説明・要件メモ等）からフローチャートを作成する場合、
まず以下の手順で情報を分析・整理し、ユーザーに確認を取る。

### 1-1. テキストから抽出すべき情報

提供されたテキストを読み取り、以下の観点で情報を抽出する：

| 抽出項目 | 具体的に何を見るか | 例 |
|---------|------------------|-----|
| **プロセスの流れ** | 開始トリガー → 中間ステップ → 完了条件。時系列順の業務手順 | 「受注→製造指示→出荷→請求」 |
| **関係者・部門** | 誰が（どの部門が）何をするか。レーン候補となる役割の特定 | 顧客、営業、製造、品管、経理 |
| **判断・分岐ポイント** | Yes/No や条件分岐が発生する箇所。承認・却下、合否判定など | 「在庫あり→出荷 / なし→発注」 |
| **システム連携箇所** | システム操作やデータ入力・参照が発生するポイント | 「販売管理システムで受注入力」「基幹で在庫確認」 |
| **例外・エラーフロー** | 通常フロー以外の処理（差戻し、キャンセル、やり直し等） | 「検査NG→再製造」「顧客キャンセル→返却」 |
| **並行処理** | 同時に進行する複数の作業がないか | 「見積作成と同時に在庫確認」 |
| **補足情報・業務ルール** | 付箋メモとして記載すべき注意事項や業務ルール | 「3営業日以内に回答必須」 |

### 1-2. 整理結果の出力フォーマット

抽出した情報を以下のフォーマットでユーザーに提示する：

```
【ワークフロー分析結果】

■ ワークフローの概要
  [1-2文でフローの全体像を記述]

■ 識別できたプロセス（時系列順）
  1. [ステップ名] → 担当: [部門/役割]
  2. [ステップ名] → 担当: [部門/役割]
  ...

■ レーン構成（左から右の並び順）
  1. [レーン名] - [役割の説明]
  2. [レーン名] - [役割の説明]
  ...

■ 識別できた分岐条件
  - [分岐ポイント]: [条件] → Yes: [処理] / No: [処理]
  ...

■ システム連携
  - [システム名]: [どの処理で使用するか]
  ...

■ 例外フロー
  - [例外名]: [発生条件と処理内容]
  ...

■ 補足メモ（付箋として表示）
  - [メモ内容]
  ...

【不明確な点・確認事項】
  - [確認が必要な点をリストアップ]
```

### 1-3. ユーザー確認

分析結果を提示した後、以下のメッセージを表示する：

> 上記の分析内容でよろしければ「OK」「はい」「了解」のいずれかを入力してください。
> 調整が必要な場合はお知らせください：
> - 不明確な点について確認したい
> - プロセスを追加・削除・変更したい
> - 分岐条件を変えたい
> - レーンの構成を変えたい

**ユーザーの確認が得られてからフェーズ2に進む。**

### 1-4. フェーズ1をスキップする場合

以下の場合はフェーズ1を省略し、直接フェーズ2に進んでよい：
- ユーザーがレーン名とフロー手順を明確に指定している場合
- 既存フローの修正・追加を依頼された場合
- ユーザーが「分析不要、そのまま作って」と明示した場合

## フェーズ2：フロー作成（XMLの構築と出力）

### 2-1. ユーザーからの入力情報を整理

以下の情報を確認・ヒアリングする（フェーズ1で既に整理済みの場合はそれを使用）：

- **タイトル（ファイル名）**: 例）`○○様_打合せ資料`
- **レーン定義**: 各レーンの名前と数（可変、2〜8程度）
- **プロセスフロー**: 各ステップの内容、どのレーンに属するか、分岐条件
- **補足メモ**: 付箋（ノート）として表示する補足情報

### 2-2. XMLを構築

`references/xml-structure.md` を読み込んで、XML構造の詳細仕様に従ってXMLを生成する。

### 2-3. ファイル出力

生成したXMLを `.drawio` 拡張子でファイルとして保存し、`/mnt/user-data/outputs/` にコピーする。

---

## フェーズ3：自動検証（必須）

XML生成またはエッジ修正の後、**ユーザーに報告する前に必ず実行する**。
目視確認に頼らず、座標計算で機械的に問題を検出する。

### 3-1. 検証スクリプトの実行

生成したXMLをPythonで解析し、以下の2種類の問題を自動検出する。
`references/edge-validator.py` を参照してスクリプトを実行すること。

#### 検出項目1: 線の重なり（CROSS）
エッジの各直線セグメント（水平/垂直）が、source/target以外のノードの
バウンディングボックス（5pxマージン付き）と交差していないか検証する。

- **CROSS-H**: 水平セグメントがノードを横切る
- **CROSS-V**: 垂直セグメントがノードを縦切る

#### 検出項目2: 不要な迂回（DETOUR）
経由点の並びで、Y座標またはX座標が「行って戻る」パターンを検出する。

- **DETOUR-Y**: Y座標が下がってから上がる（またはその逆）、差分20px超
- **DETOUR-X**: X座標が右に行ってから左に戻る（またはその逆）、差分60px超
  - ※60px以下のX方向の折り返しは、ノード側面への最終アプローチとして許容

### 3-2. 修正パターン

検出された問題に対して、以下のパターンで修正する。

#### パターンE: 左側コリドール（No分岐の垂直迂回）
判断ノードのNo側が下方のノードに直進すると中間ノードを通過する場合、
レーン左端を通って迂回する。

```
判断ノード → exitX=0, exitY=0.5（左側面）
  → 経由点(レーン左端 x=レーンX+30, 判断Y)
  → 経由点(レーン左端, ターゲットY-20)
  → ターゲット entryX=0.5, entryY=0（上端中央）
```

**ポイント**: 複数のNo分岐が同じレーン内で左側コリドールを使う場合、
X座標を10pxずつずらして線の重なりを避ける（x=420, x=430, x=440...）。

#### パターンF: 右側コリドール（複数ルートの合流）
複数のエッジが1つのノードに上方から合流し、中間に他のノードがある場合、
レーンの右端外側を通って合流先の右側面に入る。

```
ソースノード → exitX=1, exitY=0.5（右側面）
  → 経由点(レーン右端外側 x=レーンX+レーン幅+10, ソースY)
  → ターゲット entryX=1, entryY=0.2（右側面上部）
```

**ポイント**: 合流先の entryY を分散させる（0.2, 0.4, 0.6, 0.8）。
右側コリドールのX座標も5pxずつずらす（x=2220, x=2215, x=2210...）。

#### パターンG: 付箋・帳票ノードの退避
付箋（note）や帳票（document）がメインフローの縦線上にある場合、
線を迂回させるのではなく**ノード自体をレーン内の右寄り（x=200〜）に移動**する。
付箋・帳票はフロー上の必須ノードではないため、位置調整で対応するのが最もシンプル。

### 3-3. 修正後の再検証

修正を適用した後、**同じ検証スクリプトを再実行**して問題が0件になることを確認する。
XML構文エラー（閉じタグ漏れ等）も合わせて検証する。
問題が残っている場合は修正→再検証を繰り返す。

### 3-4. 既存ファイルの修正手順

既存の `.drawio` ファイルを修正する場合の手順：

1. **バックアップ**: 元ファイルをコピーしておく（`- コピー.drawio`）
2. **シート特定**: `grep -n '<diagram' file.drawio` でシート行範囲を確認
3. **シート抽出**: `sed -n 'START,ENDp'` で対象シートを抽出し内容を確認
4. **新シート生成**: 新しいXMLを生成（`<diagram>〜</diagram>`のみ）
5. **シート置換**: `head -N | cat new.xml | tail -n +M` で元ファイルに組み込み
6. **整合性検証**: `grep -c '<diagram'` でシート数、`grep -c '</diagram>'` で閉じタグ数が一致
7. **自動検証**: フェーズ3の検証スクリプトを実行

## 設計原則（厳守）

### レイアウト構造
- スイムレーン構造を採用し、役割や部門ごとにレーンを分ける（縦方向レーン使用）
- システム（販売管理システム等）は独立レーンにせず、操作する部門のレーンに含める
- **上から下への一方通行の流れ** で構成（上に戻る矢印は禁止）
- 縦幅は必要に応じて拡張可能（制限なし）
- **自動レイアウト（Stack Layout等）は絶対に使用しない**（手動調整の余地を残す）

### 座標配置ルール
- 同一レーン内のプロセスは、Y座標を **最低100px** ずつ空けて配置（詰まりによる線の重複を防止）
- 隣接レーンへの移動がある場合、Y座標を揃えて水平位置を合わせるのを基本とする
- 各レーンの幅は最低 **240px** 以上確保し、迂回ルートのスペースを設ける

### 矢印・接続線（最重要）
- **直交線の強制**: `edgeStyle=orthogonalEdgeStyle` を必ず使用
- **接続ポイントの原則**: 上から入り（entryY=0）、下から出る（exitY=1）
- 条件分岐やループバックの場合のみ、側面（exitX=0 or 1）を使用可
- **矢印が他のボックスの上を通過することを絶対に禁止**（これを守るためにすべてのエッジで経由点を計画する）
- 遠くのレーンに接続する場合は `<Array as="points">` で明示的な経由点を記述
- 複数線が合流する場合、接続先の `entryX` を微調整して矢印の先端が一点に集中しないようにする

### 線の重なり回避ルール（厳守）

**全エッジに対して、ノードとの重なりがないか検証し、重なる場合は経由点で迂回すること。**

#### パターンA: 判断ノードのNo分岐でスキップ（同レーン内）
判断のNo側が下方の離れたノードに接続する場合、中間ノードを迂回する。

```
判断ノード (y=1000)
  ├─ Yes(下) → exitX=0.5,exitY=1 → 直下のノード (y=1200)
  └─ No(横)  → exitX=1,exitY=0.5 → 経由点(レーン右端, y=1000)
                                    → 経由点(レーン右端, y=1500)
                                    → ターゲット (y=1500, entryX=0.8)
```

**XMLイメージ:**
```xml
<mxCell id="edge_no" value="No"
    style="edgeStyle=orthogonalEdgeStyle;...exitX=1;exitY=0.5;entryX=0.8;entryY=0;"
    parent="1" source="dec_xxx" target="proc_yyy" edge="1">
    <mxGeometry relative="1" as="geometry">
        <Array as="points">
            <mxPoint x="[レーンX+レーン幅-10]" y="[判断のグローバルY+40]"/>
            <mxPoint x="[レーンX+レーン幅-10]" y="[ターゲットのグローバルY-20]"/>
        </Array>
    </mxGeometry>
</mxCell>
```

#### パターンB: レーン跨ぎ（横方向→下方向のL字）
別レーンのノードに接続する場合、必ず経由点でL字またはコの字に曲げる。

```
レーン2のノード (y=1000) → exitX=1,exitY=0.5
  → 経由点(レーン2とレーン3の間, y=1000)
  → 経由点(レーン3の中央X, y=1000)
  → レーン3のノード (y=1200, entryX=0.5,entryY=0)
```

**重要:** 経由点のY座標は、経路上にある他のノードのY範囲を避ける値にすること。

#### パターンC: 複数ルートの合流
3本以上のエッジが1つのノードに合流する場合、entryXを分散させる。

```
ルートA → 合流ノード (entryX=0.25, entryY=0)
ルートB → 合流ノード (entryX=0.5,  entryY=0)
ルートC → 合流ノード (entryX=0.75, entryY=0)
```

合流元が異なるレーンにある場合は、各ルートの経由点のX座標も分離する。

#### パターンD: 帳票オブジェクトへの接続
帳票（document shape）はメインフローの横に配置し、接続は**破線（dashed=1）**で横方向に引く。

```
プロセスノード → exitX=1,exitY=0.5 → 帳票 (entryX=0,entryY=0.5)
```

帳票への線がメインフローの縦線と交差しないよう、帳票はレーン内の右寄りまたは左寄りに配置する。

#### 経由点の座標計算チェックリスト
エッジを定義する際、以下を確認すること:
1. **出発点→経由点1**: この直線上に他のノードがないか？
2. **経由点1→経由点2**: この直線上に他のノードがないか？
3. **最終経由点→到着点**: この直線上に他のノードがないか？
4. 重なりがある場合 → 経由点を追加してノードの外側を通るルートに変更

### 条件分岐の可視化
- 菱形（判断ノード）から出る矢印には **必ずラベルを設定**（`value="Yes"` / `value="No"` / `value="[条件内容]"`）
- **Main Flow (Yes/通常)**: 下方向へ進む（exitX=0.5, exitY=1）
- **Alternative (No/例外)**: 右または左方向へ進む（exitX=0 or 1, exitY=0.5）
- **No分岐が下方のノードに接続する場合**: パターンAの迂回ルールを必ず適用する

### 視覚的要素
- 役割別・機能別に色分けを実施（淡いパステルカラー推奨）
- フォントサイズ：タイトル20pt以上、本文12pt以上
- 重要な情報（特典、新機能など）は別枠で最下部または右側の空きスペースに配置

## エッジ（接続矢印）ルール

- 基本スタイル: `edgeStyle=orthogonalEdgeStyle;rounded=0;orthogonalLoop=1;jettySize=auto;html=1;`
- ラベル付きエッジ: `value` 属性にラベルテキストを設定
- 差戻しなど逆方向フロー: `dashed=1` を追加
- レーン間の接続: `exitX/exitY/entryX/entryY` で接続方向を制御
- 経由点が必要な場合: `<Array as="points">` で中間点を指定

## 座標計算ルール

### レーンの配置
- レーンは **横方向に並列配置**（左から右）
- 開始位置: x=40, y=40
- 各レーンの幅: 240〜400px（内容に応じて調整）
- 各レーンの高さ: 全レーン統一（最長のフローに合わせる。デフォルト1600px）

### ノードの配置
- ノードの座標は **所属レーン内のローカル座標**
- レーン内の左マージン: 20〜60px
- ノード間の縦間隔: **最低100px**（判断ノードの前後は**150px以上**）
- フローの進行方向: 上から下
- **帳票オブジェクトはメインフローの横に配置**（レーン内の右寄り or 左寄り）。メインフローの縦線と重ならない位置にする
- レーン右端に10〜20pxの「迂回スペース」を確保し、No分岐の迂回線がノードと重ならないようにする

### エッジの座標
- エッジの source/target は **ノードIDで指定**
- レーンを跨ぐエッジの中間点は **グローバル座標（parent="1"）**
- **すべてのレーン跨ぎエッジに `<Array as="points">` で経由点を明示設定する**（自動ルーティングに頼らない）
- 経由点のY座標は、経路上にある他のノードのY範囲（y〜y+height）と重ならない値にすること
- 経由点のX座標は、経路上にある他のノードのX範囲（x〜x+width）と重ならない値にすること

## 重要な注意事項

1. **IDの命名規則**: ノードIDは内容を反映した英字スネークケースを使用（例: `proc_sales_input`, `dec_mfg_action`）
2. **parent属性**: レーン内ノードは `parent="レーンID"`、エッジは `parent="1"`（ルート）
3. **改行**: テキスト内改行は `&#xa;` を使用。HTMLタグ `<br>` も使用可
